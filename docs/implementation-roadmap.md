
# 游戏实现路线图（Roadmap）

本文档用于固化项目的模块化实现顺序，遵循“先完整封版一个模块，再进入下一个模块”，以降低耦合与返工成本。

---

## 1. 执行原则

1. 模块串行封版：同一时间只允许一个主模块处于开发中。  
2. 接口先行：跨模块只通过`PlayerData`、`DataManager`、`EventBus`、`GameManager`交互。  
3. 先闭环后扩展：先打通可玩闭环，再增加系统深度和内容量。  
4. 每模块有明确DoD（Definition of Done）：满足DoD才允许进入下一模块。  
5. 配置优先：角色、技能、装备、敌人尽量配置驱动，减少硬编码。

---

## 2. 模块顺序总览

1. 模块0：工程底座与数据契约  
2. 模块1：角色与技能配置系统  
3. 模块2：准备系统（英雄部位装备）  
4. 模块3：战斗系统完整化（ATB + 结算）  
5. 模块4：探索系统完整化（节点/事件/SAN）  
6. 模块5：枢纽经营系统  
7. 模块6：养成与长期进度系统  
8. 模块7：内容装填与上线前打磨

---

## 3. 分模块计划

## 模块0：工程底座与数据契约

目标：统一基础框架，减少后续模块反复改底层。

主要工作：
1. 收口全局状态流：`MAIN_MENU -> HUB -> EXPEDITION -> COMBAT`。  
2. 统一静态数据加载与错误提示规范。  
3. 定义存档版本迁移规则（旧档兼容）。  
4. 固化关键信号命名与触发时机。  

DoD：
1. 新开档、读档、场景切换稳定可用。  
2. 缺失配置可定位到具体文件与ID。  
3. 存档版本升级不丢关键字段。  

依赖文档：`docs/architecture.md`

---

## 模块1：角色与技能配置系统

目标：角色初始属性、成长、技能全部配置驱动。

主要工作：
1. 固化`CharacterData/SkillData`字段与默认值。  
2. 固化角色运行时构建与旧档字段补齐。  
3. 建立技能配置目录与示例模板。  
4. UI显示改为读取“计算后属性”。  

DoD：
1. 新增角色/技能仅改`data/characters`与`data/skills`即可生效。  
2. 旧存档角色可自动补齐新字段。  
3. 角色卡与战斗入口读取配置一致。  

依赖文档：`docs/combat-attribute-design.md`

---

## 模块2：准备系统（英雄部位装备）

目标：出征前可完整完成编队与换装，主角装备仅预留结构。

主要工作：
1. 实现英雄部位装备槽与穿戴规则。  
2. 实现背包与装备联动（装上/卸下/替换）。  
3. 增加准备面板并接管“出征前确认”。  
4. 主角装备数据结构预留，不参与数值。  

DoD：
1. 英雄每个部位可独立装备，属性实时刷新。  
2. 同一装备不会被多角色重复占用。  
3. 存档读写后装备状态不丢失。  

依赖文档：`docs/preparation-system-design.md`

---

## 模块3：战斗系统完整化

目标：战斗模块可独立稳定运行并可调参。

主要工作：
1. 实现ATB循环与行动队列。  
2. 实现命中、暴击、伤害、治疗、状态结算。  
3. 实现异化值积累、失控、战后衰减。  
4. 实现战斗胜负、掉落、资源回写。  

DoD：
1. 战斗场景可独立跑通完整开战到结算。  
2. 核心数值参数可通过配置调整。  
3. 连续多场战斗无状态泄漏与卡死。  

依赖文档：`docs/combat-attribute-design.md`、`docs/play-design.md`

---

## 模块4：探索系统完整化

目标：探索流程从进入黑门到返回枢纽形成完整外循环。

主要工作：
1. 固化地图节点推进、迷雾揭示、移动消耗。  
2. 固化事件分支与SAN分段文本表现。  
3. 规范战斗节点/BOSS节点与战斗模块对接。  
4. 实现撤退、崩溃、探索奖励回写。  

DoD：
1. 单次探索可从入口走到退出并正确结算。  
2. SAN和精神链接对事件与战斗影响可验证。  
3. 资源消耗与收益可闭环回流到枢纽。  

依赖文档：`docs/play-design.md`

---

## 模块5：枢纽经营系统

目标：内循环（生产/分配/升级）稳定支撑探索与战斗。

主要工作：
1. 设施升级、工人分配、产出计算。  
2. 招募入口与资源约束接入。  
3. 枢纽UI状态刷新与事件反馈。  
4. 自动保存点与关键提示优化。  

DoD：
1. 资源产出-消耗链可持续运行。  
2. 枢纽行为能明显影响探索准备与结果。  
3. 无“资源死锁”与关键按钮无效问题。  

依赖文档：`docs/play-design.md`、`docs/game-design.md`

---

## 模块6：养成与长期进度系统

目标：形成中长期目标与门槛推进。

主要工作：
1. 角色经验、等级、星级成长。  
2. 转职与技能树（先做基础分支）。  
3. 装备成长（强化或重铸二选一先落地）。  
4. 黑门解锁条件与难度曲线。  

DoD：
1. 完整跑通“打本 -> 养成 -> 再挑战”的成长闭环。  
2. 数值曲线无明显断层。  
3. 第一扇黑门到第二扇黑门过渡平滑。  

依赖文档：`docs/game-design.md`

---

## 模块7：内容装填与上线前打磨

目标：形成可测试版本与可持续扩展内容管线。

主要工作：
1. 填充章节事件、敌人、叙事碎片。  
2. 完善UI反馈、音效、日志可读性。  
3. 性能与稳定性优化（移动端目标机型）。  
4. 平衡性首轮调参与测试修复。  

DoD：
1. 至少“第一扇黑门”达到可外测质量。  
2. 核心流程无阻塞，崩溃率可控。  
3. 主要系统参数具备可维护的配置表。  

依赖文档：`docs/background.md`、`docs/game-design.md`

---

## 4. 里程碑

1. M1（模块0-3完成）：战斗可玩版（有准备、有战斗、有结算）。  
2. M2（模块0-5完成）：完整单局循环（Hub -> Prep -> Expedition -> Combat -> Hub）。  
3. M3（模块0-7完成）：第一扇黑门完整版（可外部测试）。

---

## 5. 模块切换门禁（必须满足）

1. 当前模块DoD全部通过。  
2. 已知P0/P1缺陷清零，仅允许保留低优先级体验问题。  
3. 关键数据结构冻结并更新文档。  
4. 至少完成一次“从主菜单到返回枢纽”的完整冒烟验证。

---

## 6. 当前执行起点

当前建议从模块2开始（模块0和模块1已具备基础实现），按以下顺序推进：

1. 落地英雄部位装备数据结构与`equipment_service`。  
2. 落地准备面板与出征前确认流程。  
3. 完成后再进入模块3战斗完整化。
